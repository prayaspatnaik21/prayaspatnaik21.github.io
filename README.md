# Prayas Patnaik - Portfolio Website

This is a personal portfolio website hosted with GitHub Pages.

The site is mostly plain HTML, CSS, and JavaScript. Jekyll is used only for the blog so posts can be written as Markdown files instead of hand-written HTML pages.

Live site:

```text
https://prayaspatnaik21.github.io
```

## Big Picture

Think of the repo in two parts:

```text
Static portfolio pages
index.html, work.html, css/, js/, images/, assets/

Jekyll blog system
blog.html, _posts/, _drafts/, _layouts/, _config.yml, Gemfile
```

The final website that visitors see is still static HTML/CSS/JS. Ruby and Jekyll are build tools. They turn Markdown blog posts into normal HTML pages.

```text
Markdown posts in _posts/
        +
Layouts in _layouts/
        +
Settings in _config.yml
        |
        v
Jekyll build
        |
        v
Static pages served by GitHub Pages
```

## What Each Part Does

### `index.html`

This is the homepage. Edit this when you want to change:

- Name, headline, or contact links
- Profile image section
- "What I Do"
- Professional experience shown on the homepage
- Education
- Homepage call-to-action buttons
- Navigation links on the homepage

Important related files:

- Styling comes from `css/styles.css`
- Dark mode comes from `js/dark-mode.js`
- Greeting/profile animation comes from `js/script.js`
- Original profile photo is `images/prayas.jpg`
- Rendered homepage profile images are `images/prayas-profile-420.jpg` and `images/prayas-profile-840.jpg`

### `work.html`

This is the detailed work experience page. Edit this when you want to change:

- Job titles
- Company names
- Work dates
- Bullet points
- Detailed professional experience

This page is plain HTML. No Jekyll loop is involved.

### `blog.html`

This is the blog index page. It uses Jekyll to automatically list posts from `_posts/`.

Usually you should not manually add blog cards here. Instead, add a Markdown file to `_posts/`, and Jekyll will list it automatically.

The important loop is:

```liquid
{% for post in site.posts %}
```

That means every valid Markdown post in `_posts/` becomes a blog card.

The page is published at:

```text
/blog.html
```

### `_posts/`

This folder contains published blog posts.

Every published post must use this filename format:

```text
YYYY-MM-DD-short-title.md
```

Example:

```text
_posts/2026-06-28-sharpening.md
```

The date prefix matters. Jekyll uses it to recognize the file as a blog post.

### `_drafts/`

This folder contains unpublished notes. Drafts are useful for rough posts that are not ready for the public site.

Draft files do not need a date in the filename.

Example:

```text
_drafts/mtf.md
```

Drafts do not appear on the live GitHub Pages site unless they are moved into `_posts/`.

### `_layouts/default.html`

This is the shared wrapper for Jekyll-rendered pages.

It contains:

- HTML document structure
- Jekyll SEO tag
- Navigation for Jekyll pages
- Footer for Jekyll pages
- CSS and JS links for Jekyll pages

Blog pages use this layout indirectly.

### `_layouts/post.html`

This controls the individual blog post page design.

Edit this when you want to change:

- Blog post title layout
- Date/category/reading time display
- Post body wrapper
- Previous/next post links

### `_config.yml`

This is the Jekyll configuration file.

It controls:

- Site title
- Site description
- Author/email
- Site URL
- Markdown engine
- Blog post permalink format
- Jekyll plugins
- Files/folders excluded from the build
- Default layout for posts

Important line:

```yaml
permalink: /blog/:title/
```

This makes a post like:

```text
_posts/2026-06-28-sharpening.md
```

publish at:

```text
/blog/sharpening/
```

### `css/styles.css`

This is the main stylesheet for the site.

Edit this when you want to change:

- Colors
- Spacing
- Fonts
- Cards
- Navigation styling
- Blog card styling
- Blog post styling
- Mobile responsive layout
- Dark mode theme variables

The color variables are near the top:

```css
:root {
    --bg-primary: #ffffff;
    --bg-secondary: #fffef5;
    --text-primary: #1a1a1a;
}
```

Dark mode variables are under:

```css
body.dark-mode {
    ...
}
```

### `js/dark-mode.js`

This controls the dark mode toggle.

It:

- Detects the saved theme from `localStorage`
- Applies/removes `body.dark-mode`
- Updates the button icon

### `js/script.js`

This is for homepage interactions.

It currently supports the profile/greeting animation and small client-side behavior.

### `assets/`

Use this folder for downloadable files.

Current file:

```text
assets/resume.docx
```

If you replace the resume, keep the same filename or update all navigation links that point to it.

### `images/`

Use this folder for site images.

Current file:

```text
images/prayas.jpg
```

If you replace the profile photo, keep `images/prayas.jpg` as the original source and regenerate the optimized profile crops:

```text
images/prayas-profile-420.jpg
images/prayas-profile-840.jpg
```

The homepage uses `srcset` in `index.html` so high-DPI screens get the sharper 840px crop.

## Add A New Blog Post

Create a new Markdown file in `_posts/`.

Filename format:

```text
_posts/YYYY-MM-DD-short-title.md
```

Example:

```text
_posts/2026-07-01-camera-exposure-notes.md
```

Start the file with front matter:

```yaml
---
layout: post
title: Camera Exposure Notes
date: 2026-07-01
category: "Image Processing"
reading_time: 6
excerpt: "Notes on exposure control, image brightness, and tuning tradeoffs."
---
```

If the post contains LaTeX-style math, add:

```yaml
math: true
```

Then you can write inline math:

```markdown
$L_{sharp}(x)$
```

or display math:

```markdown
$$
L_{sharp}(x) = L(x) - \frac{k_{sharp}}{2}
\left(L(x - v) + L(x + v)\right)
$$
```

Then write the post in Markdown:

```markdown
## Introduction

Write your post here.

## Notes

- One idea
- Another idea
```

After pushing, the post will appear automatically on:

```text
https://prayaspatnaik21.github.io/blog.html
```

And the post URL will look like:

```text
https://prayaspatnaik21.github.io/blog/camera-exposure-notes/
```

## Publish A Draft

Move or copy a draft from `_drafts/` to `_posts/`.

Example:

```text
_drafts/mtf.md
```

becomes:

```text
_posts/2026-07-01-mtf.md
```

Make sure the file has front matter:

```yaml
---
layout: post
title: How to Read MTF
date: 2026-07-01
category: "Image Processing"
reading_time: 12
excerpt: "Notes on point spread functions, modulation transfer, and edge definition."
---
```

## Local Preview

Use Jekyll for local preview.

Python's HTTP server is fine for plain HTML pages, but it does not run Jekyll. That means Python will not correctly render:

- `blog.html`
- `_posts/`
- `_layouts/`
- Liquid tags like `{% for post in site.posts %}`

Recommended:

```bash
bundle install
bundle exec jekyll serve
```

Open:

```text
http://localhost:4000
```

On this Mac, the default `ruby`/`bundle` may point to Homebrew Ruby 4.x, which is too new for the current GitHub Pages dependency set. Homebrew Ruby 3.2 is installed and works with this repo.

Use this command if the generic `bundle exec jekyll serve` fails:

```bash
BUNDLE_PATH=vendor/bundle /opt/homebrew/opt/ruby@3.2/bin/bundle exec /opt/homebrew/opt/ruby@3.2/bin/ruby -S jekyll serve --host 127.0.0.1 --port 4001
```

Then open:

```text
http://127.0.0.1:4001/
```

Jekyll will rebuild when files change. Refresh the browser to see the latest output.

Preview drafts too:

```bash
BUNDLE_PATH=vendor/bundle /opt/homebrew/opt/ruby@3.2/bin/bundle exec /opt/homebrew/opt/ruby@3.2/bin/ruby -S jekyll serve --drafts --host 127.0.0.1 --port 4001
```

This repo includes:

```text
.ruby-version
```

set to:

```text
3.2.11
```

Use Ruby 3.x for local Jekyll builds. GitHub Pages' current Jekyll dependency set is not compatible with Ruby 4.x.

Verified locally:

```text
Ruby: 3.2.11 from /opt/homebrew/opt/ruby@3.2/bin/ruby
Preview URL: http://127.0.0.1:4001/
Verified pages: /, /blog.html, /blog/sharpening/
```

## Troubleshooting

### Blog page shows 404

Check:

- `blog.html` exists at the repo root.
- `blog.html` has `permalink: /blog.html` in its front matter.
- GitHub Pages build completed successfully.
- Browser/CDN cache has had a few minutes to refresh.

### New post does not appear

Check:

- File is inside `_posts/`.
- Filename starts with a date: `YYYY-MM-DD-title.md`.
- File has front matter at the top.
- The post date is not accidentally in the future.
- You committed and pushed the file.

### Python server shows weird Liquid text

That is expected. Python does not run Jekyll.

Use:

```bash
BUNDLE_PATH=vendor/bundle /opt/homebrew/opt/ruby@3.2/bin/bundle exec /opt/homebrew/opt/ruby@3.2/bin/ruby -S jekyll serve --host 127.0.0.1 --port 4001
```

### Local Jekyll fails with Ruby version errors

Use Ruby 3.x. This machine has a working Ruby 3.2 install at `/opt/homebrew/opt/ruby@3.2/bin/ruby`.

Ruby is only a local build tool here. The browser still receives plain HTML/CSS/JS.
